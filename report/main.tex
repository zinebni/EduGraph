\begin{abstract}
Designing comprehensive educational curricula that align with rapidly evolving industrial standards is a critical challenge in modern software and technology education. Traditional methods of course design are slow, manual, and often fail to bridge the skill gap between university syllabi and market realities. This report presents \textbf{EduGraph}, an advanced multi-agent educational content design and orchestration platform. EduGraph leverages a team of cooperative, specialized AI agents orchestrated via \textbf{LangGraph} to research, structure, write, and enrich course curricula in real time. 

The core system is implemented through a single, comprehensive generation pipeline consisting of an orchestrator , a market-aware searcher , an educational architect , and an assessment engine . By streaming state transitions and agent messages over WebSockets, the platform delivers a highly responsive, live-rendered learning structure directly to the user. Features such as setting configuration management, generation history persistence, and interactive self-assessment modules are supported via a scalable Next.js and FastAPI architecture. We detail the system design, the coordination protocols between language models, the data schemas, and evaluate the pedagogical framework's feasibility.
\end{abstract}

\chapter{Introduction}
\label{ch:introduction}

\section{Context and Motivation}
In the era of rapid digital transformation, technological frameworks, programming languages, and industry standards evolve faster than ever before. Traditional academic institutions and online learning platforms face a major hurdle in keeping up with these shifts. A typical software engineering syllabus designed today can easily become outdated within two years.

To combat this, curriculum developers and educators spend a significant portion of their time researching trending skills, reading documentation, formulating logical progressions, writing reading summaries, and designing assessment materials such as multiple-choice questions (MCQs) and programming exercises. Automating this pipeline while retaining pedagogical quality, structural logic, and industrial relevance is an open challenge in Educational Technology (EdTech).

\section{Problem Statement}
Despite recent advancements in Large Language Models (LLMs), generating cohesive, high-quality, and structurally consistent educational courses from a single text prompt remains challenging for the following reasons:
\begin{itemize}
    \item \textbf{Context Window and Hallucinations:} Monolithic prompts passed to a single LLM to generate an entire curriculum often result in repetition, missing modules, incorrect hour allocation, and flat summaries instead of deep educational material.
    \item \textbf{Static Training Data:} LLMs are pre-trained on static datasets and cannot accurately identify the latest versions of libraries, real-world skill gaps, or emerging tooling (e.g., specific framework changes in 2026).
    \item \textbf{Lack of Structured Interactivity:} Creating functional interactive assessments (like structured JSON templates containing MCQs and hands-on coding exercises) requires strict schema validation that standard conversational LLMs regularly fail to respect when producing bulk text.
\end{itemize}

\section{Objectives}
The primary objective of this project is to build and evaluate \textbf{EduGraph}, a cooperative multi-agent educational assistant designed to solve these limitations by:
\begin{enumerate}
    \item Distributing the workload among specialized AI agents with clear, dedicated roles (searching, planning, content writing, and quiz formulating).
    \item Orchestrating agent coordination via a shared, graph-based state mechanism using LangGraph to allow multi-turn adjustments and sequence transitions.
    \item Providing real-time streaming feedback to users via a WebSocket protocol, enabling them to inspect agent thoughts, search findings, and output blocks.
    \item Interfacing the backend agentic platform with a modern, reactive web interface using Next.js, allowing configuration persistence and local SQLite history tracking.
\end{enumerate}

\section{Outline of the Report}
The rest of this report is organized as follows: Chapter~\ref{ch:literature} reviews the theoretical framework of multi-agent orchestration, state sharing, and pedagogical standards. Chapter~\ref{ch:design} details the system design, the agent workflow topology, and the database schema. \chapter{Theoretical Background \& Related Work}
\label{ch:literature}

\section{Large Language Models in Education}
LLMs such as the Google Gemini family have transformed text generation, language translation, and code synthesis~\cite{vaswani2017attention}. In education, they are used to write summaries, translate slides, and act as personalized tutors. However, traditional LLM applications are linear: a user inputs a query, and the model outputs a response. This does not align with the structured, iterative nature of curriculum design, which requires checking sources, revising plans, and formatting modules.

\section{Multi-Agent Systems (MAS)}
Multi-Agent Systems consist of multiple autonomous entities (agents) that interact with each other to achieve collective goals~\cite{wooldridge2009introduction}. In contrast to single-agent systems, MAS breaks complex tasks into small, manageable domains assigned to specialized agents. These agents can use specific tools (such as search engines or databases) and communicate to pass findings, critique drafts, or coordinate sequential steps~\cite{multiagentsystems2020}.

\section{LangGraph and State-Sharing Architectures}
Historically, multi-agent frameworks (such as Autogen or CrewAI) relied on conversational chat loops or rigid linear chains. LangGraph introduces a graph-based state-sharing paradigm~\cite{langgraph2024}. A LangGraph workflow is modeled as a directed graph where:
\begin{itemize}
    \item \textbf{Nodes} represent agents, python functions, or tool executions.
    \item \textbf{Edges} define routing control flow based on conditional logic.
    \item \textbf{State} is a shared, append-only or overwrite memory tracker that allows nodes to read past execution histories and edit variables concurrently.
\end{itemize}
This state-sharing design provides precise control over multi-agent workflows, making it ideal for deterministic educational generation pipelines.

\section{Pedagogical Alignments and Bloom's Taxonomy}
To construct effective learning plans, content must be structured progressively according to established educational theories. \textit{Bloom's Taxonomy}~\cite{bloom1956taxonomy} classifies cognitive learning objectives into six hierarchical levels:
\begin{enumerate}
    \item \textbf{Remember:} Recall facts and basic concepts.
    \item \textbf{Understand:} Explain ideas or concepts.
    \item \textbf{Apply:} Use information in new situations (exercises).
    \item \textbf{Analyze:} Draw connections among ideas.
    \item \textbf{Evaluate:} Justify a stand or decision.
    \item \textbf{Create:} Produce new or original work.
\end{enumerate}
EduGraph addresses these levels by tasking the \texttt{curriculum\_writer\_agent} to build conceptual materials (Remember and Understand) and the \texttt{quiz\_exercise\_agent} to write MCQs and hands-on coding challenges (Apply, Analyze, and Create).

\chapter{System Design and Architecture}
\label{ch:design}

\section{System Overview}
EduGraph is built upon a standard three-tier architecture split into:
\begin{enumerate}
    \item \textbf{Presentation Layer (Frontend):} A Next.js application that renders inputs, handles settings, tracks agent progress, and displays the generated course.
    \item \textbf{Application Layer (Backend):} A FastAPI web server handling REST API routes (settings, history logs) and a bi-directional WebSocket interface to stream multi-agent logs.
    \item \textbf{Execution Layer (Agent Engine):} The LangGraph orchestration engine that compiles specialized agent prompts, integrates Gemini models, and runs Tavily web tools.
\end{enumerate}



\section{Agent Workflow Design}
The multi-agent graph coordinates the execution of three specialized nodes under a central supervisor:
\begin{enumerate}
    \item \textbf{Supervisor (\texttt{team\_supervisor}):} The routing orchestrator. It receives the user prompt, determines the execution sequence (typically Search $\rightarrow$ Write $\rightarrow$ Quiz/Exercise), invokes the appropriate agent, compiles findings, and determines completion.
    \item \textbf{Search Agent (\texttt{search\_agent}):} The market trend researcher. Equipped with the Tavily Search Tool, it queries the live internet for recent framework updates, standard methodologies, and skills gaps, outputting a formatted Markdown search summary.
    \item \textbf{Curriculum Writer Agent (\texttt{curriculum\_writer\_agent}):} The structural designer. It ingests search reports and outputs a structured curriculum plan, outlining lesson hours, timings, and chapters.
    \item \textbf{Quiz \& Exercise Agent (\texttt{quiz\_exercise\_agent}):} The assessment builder. It reads the written curriculum and outputs a strictly formatted JSON array containing interactive quizzes and exercises mapping to each chapter.
\end{enumerate}

\chapter{Technical Implementation}
\label{ch:implementation}

\section{Backend Infrastructure}
The backend server runs on Python 3.10+ using \textbf{FastAPI}. We use LangChain's google genai interface (\texttt{ChatGoogleGenerativeAI}) to run Gemini models. The multi-agent workflow is constructed programmatically as follows:

\begin{lstlisting}[language=Python, caption=Dynamic build\_graph function in backend/app/agents/graph.py.]
from langchain_google_genai import ChatGoogleGenerativeAI
from langgraph.prebuilt import create_react_agent
from langgraph_supervisor import create_supervisor
from app.tools.tavily_search import create_tavily_tool
from app.agents.supervisor import SUPERVISOR_PROMPT
from app.agents.search_agent import SEARCH_AGENT_PROMPT
from app.agents.curriculum_writer import CURRICULUM_WRITER_PROMPT
from app.agents.quiz_exercise import QUIZ_EXERCISE_PROMPT

def build_graph(google_api_key: str, tavily_api_key: str, model_name: str = "gemini-3.1-flash-lite"):
    # 1. Create LLM instance
    llm = ChatGoogleGenerativeAI(
        model=model_name,
        temperature=0.7,
        google_api_key=google_api_key,
    )
    
    # 2. Create tools
    tavily_tool = create_tavily_tool(tavily_api_key)
    
    # 3. Create agents
    search_agent = create_react_agent(
        model=llm, tools=[tavily_tool],
        name="search_agent", prompt=SEARCH_AGENT_PROMPT,
    )
    
    curriculum_writer_agent = create_react_agent(
        model=llm, tools=[],
        name="curriculum_writer_agent", prompt=CURRICULUM_WRITER_PROMPT,
    )
    
    quiz_exercise_agent = create_react_agent(
        model=llm, tools=[],
        name="quiz_exercise_agent", prompt=QUIZ_EXERCISE_PROMPT,
    )
    
    # 4. Create supervisor workflow
    workflow = create_supervisor(
        agents=[search_agent, curriculum_writer_agent, quiz_exercise_agent],
        model=llm,
        prompt=SUPERVISOR_PROMPT,
    )
    
    # 5. Compile and return
    return workflow.compile()
\end{lstlisting}

\section{WebSocket Streaming Implementation}
To provide a feedback loop during long running agent searches and writing tasks, the client and server exchange JSON events over WebSockets. The sequence diagram of messages is shown in Fig.~\ref{fig:websocket_sequence}.



The backend sends JSON payloads detailing agent lifecycle states:
\begin{itemize}
    \item \texttt{agent\_start}: Sent when a node starts running (e.g. `search\_agent`). The frontend updates its loading stepper.
    \item \texttt{agent\_result}: Contains intermediate Markdown data or JSON structures when an agent finishes.
    \item \texttt{error}: Carries error messages if any agent raises an exception.
    \item \texttt{complete}: Final payload triggering database completion and rendering.
\end{itemize}

\chapter{Features and System Evaluation}
\label{ch:evaluation}

\section{Feature Demonstration}
EduGraph offers three major UI pages:
\begin{enumerate}
    \item \textbf{Course Generator Page:} Accepts the target topic, total hours, skill level (Beginner, Intermediate, Advanced), target audience, and custom user queries. Clicking the generate button starts the WebSocket sequence.
    \item \textbf{Configurations (Settings):} Saves user key tokens. This allows switching LLM model names dynamically (e.g. \texttt{gemini-3.1-flash}, \texttt{gemini-3.1-flash-lite}, \texttt{gemini-3.5-pro}).
    \item \textbf{History Logs:} Loads previous generations from SQLite and displays them instantly without requiring LLM regeneration.
\end{enumerate}

\section{Agent Stepper & Progress Tracker}
During generations, users see an interactive visual stepper highlighting which agent is currently active:
\begin{enumerate}
    \item \textbf{Step 1: Web Research} (Active during \texttt{search\_agent} execution).
    \item \texttt{Step 2: Designing Syllabus} (Active during \texttt{curriculum\_writer\_agent} execution).
    \item \texttt{Step 3: Creating Quizzes} (Active during \texttt{quiz\_exercise\_agent} execution).
\end{enumerate}
This feedback keeps the user informed and prevents them from closing the page during long generations.

\chapter{Conclusion and Future Recommendations}
\label{ch:conclusion}

\section{Summary of Achievements}
We designed and implemented \textbf{EduGraph}, a multi-agent system built using LangGraph, FastAPI, Next.js, and Google's Gemini models. The system automates academic course creation from a single topic request. By splitting roles between a supervisor, searcher, writer, and quiz builder, we achieve high-quality curriculum outline structures, up-to-date industrial alignment, and interactive self-assessment questions that compile directly into a persistence database.

\section{Limitations}
\begin{itemize}
    \item \textbf{Latency:} Generative pipelines can take 30 to 50 seconds to compile, making live feedback loops essential.
    \item \textbf{API Reliability:} Dependency on external APIs (Tavily, Google AI Studio) makes the platform subject to rate limits and API service disruptions.
\end{itemize}

\section{Future Directions}
For future work, we plan to extend the platform by:
\begin{enumerate}
    \item \textbf{Code Execution Sandboxes:} Embedding a secure WebAssembly or Docker-based code execution sandbox to let users write and run code directly within coding challenges and receive instant testing feedbacks.
    \item \textbf{SCORM and LTI Integrations:} Supporting course exportation to learning management systems (like Canvas or Moodle) via SCORM packages or LTI (Learning Tools Interoperability) APIs.
    \item \textbf{Multi-User Collaboration:} Allowing multiple instructors to edit outline nodes prior to running quiz agent generators.
\end{enumerate}
